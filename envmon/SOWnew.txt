1. 全体コンセプト

リアルタイム値

外部API（1weilian）から 5分おき にサーバー側で取得。

取得したデータは JSONキャッシュに保存。

フロント（index.html）は data_api を通してそのキャッシュを読むだけ。

ページはリロードせず、タコメータ（ゲージ）だけ 5分おきに更新。

7日間グラフ

基本データは DeviceHistory（DB） から取得。

ただし、直近のスロットで DeviceHistory が追いついていない部分は、キャッシュJSONから補完する。

グラフは 倉庫内の全デバイスを個別の折れ線で表示（平均線はオプション扱い／不要なら出さない）。

2. リアルタイム取得 & キャッシュ
2-1. 取得周期

実際に外部APIを叩くのは cache_worker_dj.py の役目。

起動元は scheduler アプリのバックグラウンドスレッド。

周期は EnvSettings.cache_interval（秒）で管理。

想定初期値：300 秒（5分）

ページを誰も開いていなくても、サーバー側で5分ごとに必ず取得する。

2-2. 保存形式

保存先ディレクトリ：<BASE_DIR>/cache/

ファイル構成（現状の方式を踏襲）：

最新：device_cache_latest.json

過去：device_cache_YYYYMMDDHHMMSS.json を都度作成

保持期間：EnvSettings.cache_expire_hours

初期値を 720時間（30日） にする

期限を過ぎた device_cache_*.json は自動削除

2-3. JSON内容（1 レコード）

キャッシュ1件分の要素例：

{
  "id": "C00220200AD6",
  "name": "C00220200AD6_C2-3_07",
  "temperature": "17.5",
  "humidity": "41.4",
  "last_seen": "2025-12-07 12:33:37",
  "online": true,
  "location_id": "loc05",
  "warehouse": "B倉庫FMC"
}


※ location_id / warehouse は DeviceAssignment / Location を付与して保存。

2-4. デバイスのフィルタリング

キャッシュ自体には全デバイスをそのまま保存してOK。

ただし表示側で以下は除外：

warehouse が "未割当" のもの

name が "未登録:" で始まるもの（仕様書に合わせるなら）

3. data_api の役割（リアルタイム表示）
3-1. やること

外部APIにはアクセスしない。

代わりに device_cache_latest.json を読み込み、
そこに DeviceAssignment と Location に基づく情報がすでに付いている想定。

JSONをそのままフロントへ返す（配列）。

3-2. 更新周期（フロント）

EnvSettings.interval を envmon/index.html に埋め込み。

envmon_index.js は

setInterval(fetchRealtimeAndUpdate, intervalSeconds * 1000);


で /envmon/data_api を叩き、タコメータだけ更新。

運用上、EnvSettings.interval と EnvSettings.cache_interval は 同じ値（5分）に揃える想定。

厳密に同じでなくても動くが、仕様としては「原則同じ」で決めておくと分かりやすい。

4. タコメータ（ゲージ）仕様

表示値のソース：

/envmon/data_api から取得した最新キャッシュ。

倉庫ごとに 複数デバイスの温度・湿度の平均値 を計算し、

その平均値を

左：温度ゲージ

右：湿度ゲージ
に表示。

設定値：

温度ゲージ：0〜40℃（ゲージのスケールは固定）

湿度ゲージ：0〜100%

値は setTemp(avg.temp), setHum(avg.hum) で更新。

ページリロードなし、ゲージのみ 5分ごとに書き換え。

5. 7日間グラフ仕様（history_7days）
5-1. 基本

API：/envmon/history_7days

対象期間：直近7日間（今日を含む7日）

スタート：7日前の 00:00

エンド：現在時刻

スロット：3時間ごと

00:00, 03:00, 06:00, … 21:00 の 8 スロット/日 × 7日

5-2. データソース

第1優先：DeviceHistory（DB）

各デバイスごとに、スロット [slot_start, slot_end) の間で最初に記録された1件を採用。

補完：キャッシュJSON

あるスロットで DeviceHistory にデータがない場合、

過去30日の device_cache_*.json から該当デバイス・時刻範囲の値を探し、補填。

探す対象：

直近のファイルから逆順に見て、

last_seen が slot_start ≤ last_seen < slot_end のレコードを採用（最初に見つかったもの）。

※ こうすることで、「履歴APIの実行が1日1回なので DB が最新に追い付いていない時間帯」も、
キャッシュから直近数時間分を埋めて滑らかに表示できる。

5-3. グラフの描画仕様（Chart.js）

倉庫ごとの戻り値：

{
  "倉庫名A": {
    "labels": ["MM/DD HH:MM", ...],   // 実際は "12/01 00:00" など
    "temps": [23.1, null, ...],       // 倉庫平均（必要なら）
    "hums":  [45.0, null, ...],
    "devices": [
      { "sn": "C00220200A15_47F2-C", "temps": [...], "hums": [...] },
      ...
    ]
  },
  ...
}


実際のグラフ表示：

デバイスごとに 2 本の線（温度 / 湿度）を描画

温度：実線

湿度：破線

凡例（legend）は非表示。

X 軸：

グリッド線は 00:00 のスロット位置だけ表示。

ラベルは日付のみ表示（"12/01" など）。時間は表示しない。

Y 軸（温度）：

初回描画時に、全倉庫・全デバイスの最高温度を走査。

GLOBAL_TEMP_AXIS_MAX = ceil(maxTemp + 2) とする。

温度軸レンジは 25℃固定 → min = GLOBAL_TEMP_AXIS_MAX - 25。

この max/min はページをリロードするまで固定（リアルタイム更新でグラフY軸が動かないように）。

6. スケジューラの役割分担
6-1. scheduler アプリ

SchedulerConfig.ready() でバックグラウンドスレッドを起動。

そのスレッド内で 2種類の仕事 をする：

履歴API（DeviceHistory 用）

1日1回、EnvSettings.history_fetch_time に call_command("fetch_env_history")

これはすでに実装済みのロジックを踏襲。

リアルタイムキャッシュ更新

EnvSettings.cache_interval 秒ごとに

python envmon/cache_worker_dj.py --once を subprocess で実行。

成功時に last_cache_run を更新。

失敗時はログにエラーを出して次回まで待つ。

6-2. cache_worker_dj.py

外部APIにログイン → 全デバイス取得 → DeviceAssignment で倉庫名付与。

device_cache_YYYYMMDDHHMMSS.json と device_cache_latest.json を更新。

期限切れキャッシュの削除。

7. キャッシュは JSON か DB か？

今回の仕様では キャッシュは JSON に統一する方針で確定させます。

理由：

リアルタイム値は「監視用」であり、長期保存して分析する対象ではない。

頻繁（5分ごと）に DB INSERT するとテーブルがすぐ膨らむ。

JSONファイルなら：

外部API停止時にもそのまま読める。

30日分だけ残してローテーションできる。

既に cache_worker_dj.py が JSON ベースで動いているので作りがシンプル。

履歴分析やCSVダウンロードなど「ちゃんと残したいデータ」は、既に DeviceHistory で DB 保存するので役割分担は明確です。

8. 仕様として揃えるポイントまとめ

今回の整理で、特に「揃えないとズレる」部分はこのあたりです：

data_api は「キャッシュ読み専用」であること（外部APIは叩かない）

外部API呼び出しは cache_worker_dj.py からのみ行う

scheduler が 5分おきに cache_worker_dj.py --once を起動する

キャッシュ保持期間を 30日（720時間）にする

history_7days は DB + キャッシュで穴埋めする

グラフは「全デバイス分の線」が基本。倉庫平均線はオプション扱い

温度Y軸のレンジ仕様（25℃固定、max は初回計算のみ）を正式仕様として明記




全体方針

外部API叩き係：cache_worker_dj.py

スケジューラ係：scheduler アプリ

① 日次の履歴取得（fetch_env_history）

② 5分おきのリアルタイムキャッシュ更新（cache_worker_dj.py --once）

画面表示係：envmon/views.py + envmon_index.js

リアルタイム値は キャッシュJSONを data_api 経由で読むだけ

グラフは DB（DeviceHistory）＋キャッシュ（JSON）で欠損補完

STEP 6. history_7days にキャッシュ補完ロジックを追加

のレコードを探す。

最後に、補填後の device_series を元に倉庫平均（temps_series / hums_series）を再計算、または補正。

実装上のポイント

パフォーマンスのために：

キャッシュファイルを 一度だけ読む（ループ外）。

DB側のデータとキャッシュ側のデータの 優先順位は DB > キャッシュ とする。
