📘 温湿度監視システム（envmon） 正式仕様書（最終確定版）
1. 目的

本システムは、倉庫内に設置された温湿度デバイス（1weilian）のデータを取得し、

社内ポータル（Django）でリアルタイム表示

割当倉庫別に分類して見やすく表示

履歴APIより過去データを取得し DB 保存

最新の温湿度キャッシュを JSON として保持

することで、倉庫環境を安定的に監視できる状態を構築するものである。

また、リアルタイムデータはDBへ保存せず、API → JSONキャッシュ → 表示 という軽量設計を採用する。

6. キャッシュ仕様（cache_worker）
【目的】

外部APIへのアクセス頻度を減らし、万一の API 停止時にキャッシュで動作継続可能にするため。

【キャッシュ構造】
cache/
 ├─ device_cache_latest.json     ← ☆ 最新キャッシュ（表示用）
 └─ history/
        ├─ 2025-12-01.json
        ├─ 2025-12-02.json
        └─ （最大30日分保存）

【保存方式】

最新 API データを JSON ファイルにそのまま保存

過去キャッシュは１日１ファイルとして保存

30日経過したキャッシュは自動削除

【表示との関係】

リアルタイム表示は API から直接取得（JSONキャッシュを使わない）

ただし API 障害時の fallback としてキャッシュ読み込みを準備（後で対応可）

7. 履歴データ管理（DeviceHistory）
【用途】

数値分析

CSV ダウンロード

週報/月報作成など

【データ取得方法】

1日1回、scheduler が fetch_env_history を実行し、

各 SN の最終記録時刻を確認

そこからの増分を履歴APIで取得

DeviceHistory に保存

【DB保存項目】

sn

recorded_at（タイムゾーン考慮）

temperature

humidity

生データ JSON

8. 倉庫割当
【DeviceAssignment（DB）】

device_id → location の対応

warehouse 名は Location モデルから参照

UI はドラッグ＆ドロップで操作

【割当履歴（AssignmentHistory）】

割当変更時に記録

API評価時やキャッシュには影響しない

10. 設定(EnvSettings)
設定	用途
interval	フロント更新間隔（秒）
cache_interval	キャッシュ更新間隔
cache_expire_hours	キャッシュ保存期間（推奨720時間＝30日）
history_fetch_time	履歴API実行時刻
log_directory	CSVダウンロード保存先
log_times	ログ保存時刻（将来機能）
is_fetching_history	二重実行防止のフラグ
11. ❗ 非採用となった仕様（混乱ポイントの整理）

以下は現在の正式仕様から除外済み：

❌ リアルタイムデータを EnvLog モデルとして DB に保存

→ 使わない。モデル削除可。

❌ scheduler によるリアルタイムスナップショット取得

→ 不要。履歴は履歴APIだけが担当。

❌ index ページのページリロード方式

→ 削除済。JS更新方式のみ使用。

✔ 最終結論（明文化）

本システムは、リアルタイムデータを API から直接取得し、表示用に DB 保存は行わない。
最新データは JSON キャッシュに保存し、履歴データは履歴APIからのみ取得して DB に保存する。
これが envmon の正式な最終アーキテクチャである。
