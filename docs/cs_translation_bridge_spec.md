# 管理者用CS課題管理アプリ(Claude自動翻訳付き) 設計仕様書

- 対象: 社内アプリ「既存業務課題管理」(`cs_tasks`)と連携する、**私用Mac上で動く管理者(上長)専用アプリ**
- 目的: 社内サーバ上の課題/進捗をMac側アプリで取り込み、Claudeで翻訳・閲覧・編集し、編集結果を社内サーバへ反映する。

- ステータス: ドラフト(方針確定 / 実装前)

## 0. アプリの位置づけ(別体構成)

本仕様で作るのは、社内サーバ上の `cs_tasks` とは**別体の、Mac上で動く管理者専用アプリ**である。両者は社内LANで直結できないため、**メール2経路でのみ**連携する。役割の境界は次のとおり。

- **Mac側アプリ(主役・本仕様の中心)**: 同期メールの取り込み・Claudeによる中↔日翻訳・閲覧・編集・書き戻しメールの生成。機能の大半はここ。ローカルに自分のデータ(取り込んだ課題/進捗/翻訳/編集中の下書き)を保持し、社内アプリとは独立して動く。
- **社内側アダプタ(最小限の追加)**: Mac側アプリが「編集結果を送り返し、社内サーバ側で表示する」ためだけに、社内 `cs_tasks` に**受信処理+二言語フィールド**を小さく追加する。Macが社内LANに到達できない以上、この受け口の追加は避けられない最小依存。

> つまり「Mac上の独立した管理者アプリ + 社内側に最小の受け口アダプタを1つ追加」という構成。社内アプリ本体の作り替えは行わない。

## 1. 背景と目的

社内CSメンバーは社内アプリ「既存業務課題管理」に進捗(`ProgressUpdate`)を**中国語**で記入する。上長であるユーザー本人は、これらを**日本語**で読み、上長コメント(`SupervisorComment`)やタスク欄の修正・追加を**日本語**で行いたい。記入した日本語は**中国語へ自動翻訳**し、社内サーバ側に反映してメンバーが中国語で読めるようにする。

この作業を、ユーザーの**私用Mac上で動く管理者専用アプリ(Cowork/Claude)**で行う。Claudeが翻訳・編集の中核を担い、取得〜反映を半自動〜自動で回す。

## 2. 制約と前提(設計を規定する最重要事項)

- **私用Macは社内LAN(`10.214.80.86`)に一切到達できない。** VPNも使わない。
  → API直結・Chrome自動化(サーバ画面操作)・DB直接アクセスはいずれも**不可**。
- **Macと社内サーバを繋ぐ唯一の経路はメール。** 往路・復路ともメールで設計する。
- 現状の `cs_tasks` は**メールを送るだけ**で、**受信して反映する仕組みは無い**。
  → 復路(Mac→サーバ)の自動化には、サーバ側に「メール受信→DB反映」処理を**新設**する必要がある。これが唯一の主要な新規開発。
- 往路(サーバ→Mac)は、既存の週次レポートメール基盤(`cs_tasks/email_utils.py`・`scheduler.py`・`mailcenter`)をほぼ流用できる。

## 3. 既存実装の確認結果

- 構成: Django 5.2 + SQLite。REST API なし(素のDjango)。本番は `10.214.80.86`。
- データモデル(`cs_tasks/models.py`):
  - `Task` … 課題(title / description / client_name / owner / assignee / due_date / is_closed / is_cancelled)
  - `ProgressUpdate` … 進捗(task FK / author / content / created_at / is_closed)。**メンバーが中国語で記入**。
  - `SupervisorComment` … 上長コメント(progress FK / author / content)。**is_staff のみ記入可**。
  - `WeeklyReportMailingList` / `WeeklyReportConfig` … 週報の宛先・送信スケジュール。
- メール送信基盤:
  - `mailcenter/email_utils.py: send_html_mail()` … Aliyun SMTP(`smtp.qiye.aliyun.com:587` STARTTLS)でHTMLメール送信。
  - `cs_tasks/email_utils.py: send_weekly_report()` … 週次レポートをメーリングリストへ送信。
  - `cs_tasks/scheduler.py` … 60秒ごとtickのバックグラウンドスレッド。曜日/時刻条件で週報を自動送信。`WeeklyReportConfig` で制御。
- 書き戻し用ビュー(`cs_tasks/views.py`)は `edit_progress` / `add_comment` / `edit_comment` 等が存在するが、**いずれもセッション認証のフォームPOST**(JSON APIではない)。Macからは到達不可のため本ブリッジでは使用しない。
- 権限(`cs_tasks/permissions.py`): 上長判定は `is_superuser or is_staff`。コメント付与・クローズは上長のみ。**本ブリッジのユーザーは上長(is_staff)である前提。**

## 4. 全体アーキテクチャ:Mac側アプリ + 社内側アダプタ(メール2経路で連携)

```
【往路】 portal(社内LAN) ──同期メール──▶ ユーザーのGmail(Mac)
  対象の課題/進捗/コメントを「機械可読JSON(ID付き)」として
  メール本文に埋め込んで送信。変更発生時に即時 + 定期の保険送信。

【Mac / Cowork(Claude)】
  ① Gmailコネクタで同期メールを取得 → JSONをフォルダに保存
  ② Claudeが中国語→日本語に翻訳
  ③ レビュー画面(ライブ artifact)に原文(中)+訳(日)を一覧表示
  ④ ユーザーが日本語で:上長コメント追加 / タスク欄の修正・追加
  ⑤ Claudeが日本語→中国語に翻訳し、書き戻しJSON(操作リスト)を生成
  ⑥ HMAC署名を付与し、書き戻しメールの下書きを作成

【復路】 ユーザーのGmail(Mac) ──書き戻しメール──▶ portal専用受信箱(社内)
  ★新設★ portalがIMAPで受信箱を監視 → 差出人限定 + HMAC署名検証
  → 操作を冪等にDBへ適用(SupervisorComment追加 / ProgressUpdate編集 /
    Task修正・新規追加)。中文・日本語の両方を保存。
```

各データは既存の `SupervisorComment` / `Task` / `ProgressUpdate` にそのまま乗る。

## 5. データモデル変更(中日両保持)

社内サーバ側の日中表示トグル(既存i18n)で、課題・進捗・コメントを**日本語/中国語どちらでも表示**できるよう、両言語をDBに保持する。

方針: 既存のテキストフィールドを**表示既定(中国語)**として残し、**日本語版の companion フィールド `*_ja` を追加**する。表示時は言語に応じて出し分ける。

| モデル | 既存(中国語/原文) | 追加(日本語) |
|---|---|---|
| `SupervisorComment` | `content` | `content_ja` |
| `ProgressUpdate` | `content` | `content_ja` |
| `Task` | `title` / `description` | `title_ja` / `description_ja` |

- 追加フィールドはすべて `blank=True`(既存データ互換のため null/空許容)。
- 表示ヘルパ: `display_content(lang)` のようなメソッド/プロパティを各モデルに用意し、`lang == 'ja'` かつ `*_ja` が非空なら日本語、そうでなければ原文を返す。
- マイグレーションは追加のみ(破壊的変更なし)。

> 補足: メンバーが記入する `ProgressUpdate.content`(中国語)は、Mac側で読むための翻訳(中→日)はMac側で都度行えば足り、必ずしもサーバへ書き戻す必要はない。ただしユーザーが進捗本文を編集した場合は両言語で保存する。`content_ja` は「サーバ側でも日本語表示したい」場合のキャッシュとして任意で持つ。

## 6. 往路:サーバ → Mac(同期メール)

### 6.1 トリガー

- **即時送信**: `ProgressUpdate` / `SupervisorComment` / `Task` の保存時に Django シグナル(`post_save`)で同期メールを送る。→ 「変更→数分以内にMac反映」を実現。
- **保険の定期送信**: 既存スケジューラに相乗りし、一定間隔(例: 10分)で「前回送信以降の差分」をまとめて送る。シグナル取りこぼし・送信失敗の補償。
- 送信失敗時はリトライ。連投抑制のため短時間のデバウンス(例: 直近変更を30〜60秒バッファしてまとめる)を推奨。

### 6.2 宛先

- ユーザーのGmailアドレスを宛先に設定(専用の同期用とするのが望ましい)。
- 既存 `WeeklyReportMailingList` とは別系統の宛先設定を用意(週報とは目的が異なるため混在させない)。

### 6.3 ペイロード(本文埋め込み)

Gmailコネクタは**添付ファイルを取得できない**ため、JSONは**メール本文(プレーンテキスト)に埋め込む**。判別しやすいマーカーで囲う。

```
件名: [CS-SYNC] 2026-06-01T09:00:00+08:00 seq=42

-----CS-SYNC-BEGIN-----
{
  "type": "snapshot",
  "schema": 1,
  "seq": 42,
  "generated_at": "2026-06-01T09:00:00+08:00",
  "since": "2026-06-01T08:50:00+08:00",
  "tasks": [
    {
      "id": 123,
      "title": "...(zh)...",
      "client_name": "...",
      "assignee": "張三",
      "due_date": "2026-06-10",
      "is_closed": false,
      "progress_updates": [
        {
          "id": 456,
          "author": "李四",
          "content": "...(zh)...",
          "created_at": "2026-06-01T08:45:00+08:00",
          "comments": [
            {"id": 789, "content": "...(zh)...", "content_ja": "...(ja)..."}
          ]
        }
      ]
    }
  ]
}
-----CS-SYNC-END-----
```

- `seq` は単調増加。Mac側は `seq` で取りこぼし・重複を検知。
- `since` を付け、差分(`updated_at` ベース)を送る。初回や明示要求時はフルスナップショット。

## 7. Mac側処理(Cowork / Claude)

1. **取得**: 定時タスク(例: 数分間隔)または手動で、Gmailコネクタの `search_threads`(件名 `[CS-SYNC]`)→ `get_thread` で最新本文を取得し、マーカー間のJSONをフォルダへ保存。`seq` で新着判定。
2. **翻訳**: 進捗・コメントの中国語を日本語へ翻訳。
3. **レビュー画面(ライブ artifact)**: 課題→進捗→コメントの階層で、**原文(中)+訳(日)を並記**。各進捗に「上長コメント(日本語)」入力欄、各タスクに編集欄(タイトル/詳細/客先/担当/期限)、新規タスク追加欄を用意。開いている間は自動更新で最新を取り込む。
4. **編集**: ユーザーは日本語で入力。
5. **翻訳(逆方向)**: Claudeが日本語→中国語へ翻訳。
6. **書き戻し生成**: 操作リストJSONを組み立て、HMAC署名を付与し、Gmailコネクタの `create_draft` で**書き戻しメールの下書き**を作成。

> 送信ゲート: 公式Gmailコネクタは下書き作成のみで**メール送信機能を持たない**。これを逆手に取り、**「下書きを確認 → ユーザーが送信ボタンを押す」を最終承認ゲート**とする(DBを書き換える操作なので人手の最終確認は安全側)。完全自動送信が必要な場合は、Gmailアプリパスワードを用いたSMTP直送を別途実装(フェーズ2)。

## 8. 復路:Mac → サーバ(受信処理を新設)

### 8.1 ペイロード(本文埋め込み + 署名)

```
件名: [CS-WRITEBACK] seq=42

-----CS-WB-BEGIN-----
{
  "schema": 1,
  "nonce": "9f1c... (一意)",
  "issued_at": "2026-06-01T18:30:00+09:00",
  "ops": [
    {"op_id": "uuid-1", "action": "add_comment",
     "progress_id": 456, "content_zh": "...", "content_ja": "..."},
    {"op_id": "uuid-2", "action": "edit_progress",
     "progress_id": 456, "content_zh": "...", "content_ja": "..."},
    {"op_id": "uuid-3", "action": "edit_task",
     "task_id": 123, "fields": {"title_zh": "...", "title_ja": "...",
       "description_zh": "...", "description_ja": "...",
       "client_name": "...", "due_date": "2026-06-15"}},
    {"op_id": "uuid-4", "action": "add_task",
     "fields": {"title_zh": "...", "title_ja": "...", "client_name": "...",
       "assignee_email": "...", "due_date": "..."}}
  ]
}
-----CS-WB-END-----
-----CS-WB-SIG-----
HMAC-SHA256(base64) : 上記JSON(マーカー内の生バイト列)に対する署名
-----CS-WB-SIG-END-----
```

### 8.2 受信処理(★新規コンポーネント★)

- 形態: Django 管理コマンド `process_inbound_mail`(例)を、既存スケジューラスレッド or cron から定期実行(サーバは自分の受信箱にIMAPで到達できる)。
- 専用受信箱(Aliyun IMAP)へ接続し、未読メールを取得。
- 検証(2段):
  1. **差出人限定**: 許可した差出人アドレス以外は破棄。
  2. **HMAC署名検証(案B)**: 共有秘密鍵で本文JSONのHMAC-SHA256を再計算し、署名と一致しなければ破棄。秘密鍵はメールに乗らない。本文が1文字でも改ざんされれば不一致。
- **冪等性**: 各 `op_id` を「処理済み操作」テーブルに記録し、重複適用を防ぐ。`nonce` でメール単位のリプレイも拒否。
- 適用: 1メール=1トランザクション。各 `action` を対応モデルへ反映。
  - `add_comment` → `SupervisorComment` 作成(`content`=中文, `content_ja`=日本語, author=ブリッジ用ユーザー)。
  - `edit_progress` → `ProgressUpdate.content` / `content_ja` 更新。
  - `edit_task` / `add_task` → `Task` の各フィールド(両言語)更新・作成。
- 処理後は既読化。処理結果(成功/失敗/スキップ)をログに残す。失敗時はユーザーへ通知メール(往路)を返すと親切。

### 8.3 権限の扱い

- 受信処理はビューの `@login_required` を通らないため、**ブリッジ専用のユーザー(is_staff)** を author として用い、モデル層で直接作成する。権限チェックはHMAC+差出人限定で代替。

## 9. セキュリティ

- **HMAC共有鍵**・**SMTP/IMAP認証情報**は環境変数(`os.environ` / `.env` + python-dotenv等)に移し、**リポジトリにコミットしない**。
- 現状 `intranet_portal/settings.py` に **SMTPパスワード等が平文でコミット**されている(例: `EMAIL_HOST_PASSWORD`)。受信処理の新設で攻撃面が増えるため、この機会に**環境変数化を必須**とする。
- 受信箱は本ブリッジ専用とし、一般の問い合わせ等と混在させない。
- HMAC + 差出人限定 + nonce/op_id による冪等・リプレイ防止を最低ラインとする。

## 10. リアルタイム性・同期頻度

- 実現できるのは「**数分以内のほぼリアルタイム**」。秒単位の瞬時pushは構造上不可(経路がメールのみ)。
- 内訳: サーバ側はシグナルで**即送信**、メール配送は数秒〜1分、Mac側は定時チェック間隔(調整可)。
- Mac側はCoworkが起動・稼働中に反映。スリープ/未起動の間は溜まり、次回稼働時にまとめて反映。
- 頻度を上げるほど速いがチェック回数(負荷)が増えるトレードオフ。初期値は「往路: 変更時即時＋10分保険」「Mac取得: 数分間隔」を推奨。

## 14. 将来構想:リアルタイム連動

### 14.1 なぜメールでは"瞬時"にできないか(根本制約)

メール方式の遅延は「メール往復 + Mac側のチェック間隔」に由来する。これを取り除いて秒単位の双方向リアルタイムにするには、**Macと社内サーバの間に常時接続の通信路**が要る。問題は、Macが社内LAN(`10.214.80.86`)へ**インバウンド接続できない**こと。ただし社内サーバは**アウトバウンドのインターネット接続を持つ**(現にAliyun SMTPへ送信できている)。この非対称性が、将来案の鍵になる。

ポイント: **「どちらかが相手のポートを開けて待ち受ける」必要のない方式**(=双方がアウトバウンドで同じ中継点に繋ぐ、またはアウトバウンドでメッシュを張る)を選べば、社内ファイアウォールを触らずに常時接続を作れる。

### 14.2 案① メッシュVPN(Tailscale 等)— 推奨

社内サーバとMacの両方に Tailscale 等のノードを入れ、**両者ともアウトバウンドのみ**で暗号化されたプライベート網を張る。これでMacから `10.214.80.86` 相当へ直接到達できるようになる。

- 接続後は、社内 `cs_tasks` に**軽量なJSON API**(取得・書き戻し)を追加し、メール経路を置き換え。
- リアルタイム化は、サーバの変更時(`post_save` シグナル)に **SSE(Server-Sent Events)or WebSocket(Django Channels)** でMacへプッシュ。Mac→サーバは直接API呼び出しで即時。
- 長所: 社内の公開ポート開放・社内VPN契約が不要。アウトバウンドのみで安全。メール往復が消え、秒単位の双方向同期が可能。
- 短所/前提: 社内サーバへのソフト導入と、私用Macを社内ネットワークに論理接続することについて**会社のセキュリティポリシー上の承認**が必要。

### 14.3 案② クラウド中継(リアルタイムブローカー)— ポリシー次第の代替

社内サーバへ何も"入れたくない/入れられない"場合、**双方がアウトバウンドで繋ぐクラウドの中継点(pub/sub)**を置く。

- 例: マネージドのリアルタイムサービス(Ably / Pusher / Supabase Realtime / Firebase)や、自前VPS上のWebSocket・MQTTブローカー。
- 社内サーバは変更イベントを中継点へ publish、Macは subscribe して即受信。逆方向も同様。**どちらも待ち受け不要**(アウトバウンドのみ)。
- 長所: VPNなしで双方向リアルタイム。社内サーバはアウトバウンド先が1つ増えるだけ。
- 短所: 中継点(第三者クラウド)を経由するため、**データの取り扱い・暗号化・委託先審査**が要る。社内データを社外中継に通す可否はポリシー確認必須。エンドツーエンド暗号化(中継点では中身を見せない)を併用するのが望ましい。

### 14.4 案③ 社内VPN — IT提供が前提

会社がVPNを提供し、私用Macの接続を許可するなら、VPN接続中はMacから社内サーバへ直接到達でき、案①と同じくAPI + SSE/WebSocketでリアルタイム化できる。

- 長所: 構成は素直。社内が管理する経路。
- 短所: 私用デバイスのVPN接続可否・IT手続きに依存。常時接続ではなくVPNを繋いでいる間だけ。

### 14.5 リアルタイム化の共通メカニズム(接続路が用意できた後)

接続路(案①〜③のいずれか)が確立したら、転送方式は次に置き換える。

- **サーバ → Mac(プッシュ)**: `post_save` シグナル → イベント発行 → **SSE or WebSocket** でMacへ即時配信。Macの画面(ローカルアプリ/常時接続のビュー)が即更新。受信を契機にClaudeが翻訳を走らせる。
- **Mac → サーバ**: 直接 **JSON API**(POST)で即時反映。書き戻しメール+IMAP受信は不要に。
- メール経路は撤去せず**オフライン時のフォールバック**として残すと堅牢(接続が切れてもメールで非同期同期できる)。

### 14.6 移行ロードマップ(提案)

1. **現行(フェーズ1〜2)**: メール2経路の非同期同期(数分以内)。本仕様の主軸。
2. **準リアルタイム**: 接続路を確立(案①Tailscale を第一候補)。転送をメール→直接APIに置換。Mac側はポーリング短縮 or SSEで取得。
3. **フルリアルタイム**: サーバに SSE/WebSocket プッシュを実装し、双方向を秒単位に。メールはフォールバックとして温存。

> 設計上の含み: 本仕様のJSONペイロード(往路スナップショット/復路操作リスト)は、転送手段がメールでもAPIでも**同じデータ構造で流用できる**よう設計してある。将来リアルタイム化する際、データ契約はそのまま、運ぶ経路だけを差し替えればよい。これが移行コストを最小化する。

### 14.7 まず確認すべきこと

リアルタイム化の実現可否は技術ではなく**会社のポリシー**でほぼ決まる。先に次を確認するとよい。

- 社内サーバへ Tailscale 等のソフトを導入してよいか(案①)。
- 社内データをクラウド中継へ通してよいか(案②)。
- 私用Macを社内VPNに接続してよいか(案③)。

## 15. 将来構想:Claudeによる課題・進捗の自動起票

### 15.1 概要

ユーザーがMac上でClaudeと行った業務(Claudeで整理した業務、Claudeを使って実行した作業)を、Claudeが**自動で課題(`Task`)・進捗(`ProgressUpdate`)として社内 `cs_tasks` に起票**できるようにする。これにより、上長自身の業務もメンバーの進捗と同じ土俵で可視化・管理できる。

### 15.2 位置づけ(既存設計の延長)

これは**復路(Mac→サーバ)のデータの出どころが増えるだけ**で、書き戻しの経路・HMAC認証・冪等性・二言語保存の仕組みはそのまま流用できる。新規開発は「起票案の生成」と「復路操作の拡張」に限られる。

- 既存の復路操作 `add_task` を活用し、**`add_progress`(進捗新規。`task_id` 指定 or 新規 `Task` と同時作成)を追加**する。
- ペイロードはこれまでと同じJSON契約。転送がメールでもAPI(将来)でも同一構造で流用可。

### 15.3 仕組み案

1. **入力源**: Claudeセッションでの作業内容(ユーザーが整理した業務、Claude実行の成果・結論)。
2. **抽出・分類**: Claudeが作業を要約し、「新規課題か/既存課題への進捗か」を判定。既存課題とのマッチングは `client_name`・`title` 等で候補を提示。
3. **起票案の生成**: 日本語で課題名/詳細/進捗内容を起こし、中国語へ自動翻訳。両言語で write-back op を生成。
4. **承認ゲート(必須)**: 共有業務システムへ書き込むため、**送信前にユーザーが内容を確認・承認**(既存の「下書き→送信」ゲートと同じ)。誤起票・重複・私的メモの混入を防ぐ。
5. **反映**: 承認後、復路メール(将来はAPI)で社内アダプタへ送り、`Task`/`ProgressUpdate` を冪等に作成。

### 15.4 起票者(author)と監査性

- 起票者を**ユーザー(上長)本人**とするか、**ブリッジ/Claude専用アカウント**にするか要決定。
- いずれにせよ「**Claude起票であること**」が後から分かる印(専用author / タグ / メタ情報)を付け、区別・取り消し・監査ができるようにする。

### 15.5 トリガー(任意設定)

- セッション終了時に「起票候補」を提示。
- 「今日やった業務を起票して」等の明示指示。
- 日次サマリのスケジュール(その日のClaude業務をまとめて起票案化)。

### 15.6 新規/拡張コンポーネント

- [Mac側] 起票候補の抽出・既存課題マッチング・起票案レビューUI(既存レビュー画面に統合)。
- [社内側アダプタ] 復路操作に `add_progress` を追加。`Task`/`ProgressUpdate` 作成時に「Claude起票」印を付与。

### 15.7 留意点・要検討

- **当面は必ず承認ゲート経由**とし、完全自動起票は慎重に(誤り・重複・粒度のばらつきリスク)。
- 何を「業務」として拾うかの線引き(全セッションか、ユーザー指定分のみか)。
- 課題の粒度・重複の扱い(既存課題への進捗追記を優先し、無闇に新規課題を増やさない方針が無難)。
- 監査性: Claude起票を記録し、後から一括で確認・取り消しできる導線を用意。
